# Architecture

ReviewWrite 采用“能力联邦”，不把所有上游规则复制成一个巨型提示词。

## 四层结构

```text
Evolution    评测、作者样例、有效修改保留、回归测试
Control      Review → Plan → Rewrite → Verify
Judgment     体裁、论证、篇章、语用、自然表达、泄漏检查
Ontology     作者、读者、目的、主张、证据、来源、限定、文本、修改
```

## 能力接口

每个上游能力按同一接口接入：

- `id`：稳定名称；
- `source`：来源、版本和许可；
- `applies_when`：语言、体裁和问题类型；
- `signals`：能发现的问题；
- `exceptions`：不能机械应用的场景；
- `suggestions`：允许提出的修改类型；
- `verification`：改写后必须复核的项目。

首版以说明文件和 few-shot 表达该接口。后续可以把它实现为机器可读 registry，但不应让 registry 先于真实评测。

## 控制策略

控制器不优化单一“像不像 AI”分数。它使用多目标约束：

1. 硬约束：事实、引用、政策口径、法律义务、提示和推理泄漏；
2. 主要目标：主张与证据、体裁适配、清楚和连贯；
3. 次要目标：节奏、密度、模板化表达和格式；
4. 个性目标：作者样例中的稳定选择，而不是人为添加瑕疵。

硬约束失败时停止交付；次要风格指标不得覆盖主要目标。

## 进化策略

只保留经验证有效的变化：

- `variation`：产生少量、边界明确的修改候选；
- `selection`：用体裁、事实保持和读者目标比较候选；
- `retention`：将用户确认的修改原则、作者样例和回归案例保存为可追溯资产。

数字、事实、引用、专名和法律/政策限定属于保护项，不参与自由变异。

## 上游更新

1. 检查 `sources.lock.json` 中的上游版本；
2. 阅读上游 diff，而不是直接覆盖本项目规则；
3. 将新增能力归入接口，记录适用范围和例外；
4. 新增阳性、阴性和语义保持测试；
5. 更新锁文件和 `NOTICE.md`；
6. 通过回归后才合并。

这种方式保留上游创新，也避免上游的绝对规则破坏政策、学术或法律文本。
