# Technical commentary / 技术解读

## Reader and purpose

读者希望快速理解一个模型、工具或技术方案改变了什么，但不应被口号、参数规模或无条件预测替代。文章可以有判断和节奏，判断必须能回到机制、测量口径或具体场景。

## Review priorities

1. 区分发布方描述、作者推断和行业预测。
2. 对参数量、激活量、速度、内存、功耗等数字注明统计口径、单位、来源和适用范围。
3. 解释技术机制时，至少交代对象、动作和影响，不用“抓住了”“拨正方向”等抽象动词代替机制。
4. 控制相邻段落中的二元反转、重要性宣告、用户泛化和总结性预测；单个表达可以保留，叠加时应改为证据或条件。
5. 把“更可能”“未必”“将成为”等预测写成有场景和边界的判断，并保留不确定性。
6. 用一个具体设备、工作负载、测量结果或失败条件帮助读者判断，而不是替读者笼统概括“多数用户”的需求。

## Hard blocks

- 把未核验的参数、性能或来源写成确定事实；
- 将模型发布方的宣传语改写成独立技术结论；
- 用行业预测掩盖缺少测量对象、样本或条件；
- 泄漏提示词、隐藏推理、工具过程或编辑说明。
